# Aurora 共享部署阶段 0 定版基线

更新日期：2026-04-04

## 1. 文档目的

这份文档用于把 Aurora 从“本地单人工作台”升级到“内网共享部署”之前的关键架构决策冻结下来。

阶段 0 的目标不是一次性做完全部安全和运维能力，而是先停止继续带着不同前提往下开发。后续阶段必须以这份基线为准，不再反复讨论以下问题：

- 部署单元到底是什么
- 系统到底信任哪一个身份来源
- 角色到底有哪些以及各自能做什么
- 数据隔离到底按什么边界执行
- 哪些能力这次明确不做

这份基线适用于：

- 公司内网试点部署
- 多人共享使用
- 单个可写的 Aurora 应用实例

这份基线不适用于：

- 公网直接暴露
- 多租户 SaaS
- 多实例 active-active 负载部署

## 2. 定版结论

### 2.1 部署边界

阶段 0 的标准部署拓扑冻结为：

`公司用户 -> 内网反向代理 -> Aurora 单实例 -> 持久化数据库 / 向量库 / 文件存储`

硬性约束：

- Aurora 只能部署在内网反向代理之后，不直接裸露给公网。
- 反向代理是唯一可信入口，负责把认证结果传递给 Aurora。
- Aurora 在本阶段保持“模块化单体 + 同实例后台任务”的形态。
- 状态数据库、向量库、上传文件、隔离区、日志都必须位于持久化存储上。
- 共享部署只允许一个可写 Aurora 实例；重建、上传、聊天持久化和审计都在该实例上执行。

这意味着：

- 阶段 0 不要求立刻上分布式队列或多实例 worker。
- 但阶段 0 要求当前单实例存储不再被当作随时可丢弃的本地临时目录。
- TLS、SSO 跳转、会话建立和可信头注入都属于反向代理或公司统一身份网关，不属于 React 前端。

### 2.2 身份来源

共享部署场景下，Aurora 只信任一个身份来源：

- 公司统一 SSO / OIDC
- 其结果通过反向代理或身份网关的可信 session / header 传入 Aurora

硬性约束：

- 浏览器不能在业务 API 中自报 `user_id`、`team`、`role` 或租户身份。
- 公共业务路由必须只从服务端可信认证上下文推导当前用户。
- `AUTH_MODE=development` 仅允许用于本地开发和本地测试。
- 共享部署阶段可先以 `AUTH_MODE=trusted_header` 作为过渡实现，后续再演进到 Aurora 内部直接接 OIDC/session。

反向代理或会话层至少需要提供：

- 稳定的 `user_id`
- `display_name` 与 `email`
- `team_id`
- `role`
- 可访问的项目列表
- 可选的当前激活项目

### 2.3 权限模型

阶段 0 冻结 Aurora 的角色模型为四类，后续阶段不得再随意扩散角色语义：

- `admin`
- `operator`
- `member`
- `viewer`

推荐权限边界如下：

| 角色 | 定位 | 允许动作 | 明确禁止 |
| --- | --- | --- | --- |
| `admin` | 平台管理员 / 内部管理者 | 全部业务 API、内部校验 API、配置修改、日志清理、知识库重建/重置、provider dry-run、审计查看 | 无 |
| `operator` | 运维 / 知识库操作员 | 聊天、文档管理、图谱查看、知识库重建/扫描/同步/重置、日志查看、健康检查 | 配置写入、日志清理、内部管理员接口、身份绕过 |
| `member` | 普通项目成员 | 聊天、上传/重命名/删除/改元数据、知识库状态查看、图谱查看 | 配置修改、日志访问、重建/重置、内部接口 |
| `viewer` | 只读使用者 | 总览查看、文档预览、图谱查看、只读问答 | 文档写入、配置修改、日志访问、重建/重置、内部接口 |

硬性约束：

- 所有管理类能力必须由服务端权限检查控制，不能再依赖“带某个 header 就放行”

### 2.4 数据模型与隔离

阶段 0 冻结数据模型为：

- 单租户
- 不支持跨租户能力
- 隔离维度为 `project / team / user / session`

硬性约束：

- `tenant_id` 是部署级固定值，不是正常业务流里用户可切换的参数。
- `project_id` 是文档、会话、聊天历史、知识检索的主隔离边界。
- `team_id` 用于组织分组和审计归因。
- `user_id` 与 `session_id` 是项目内更细粒度的隔离维度。
- 公共 API 不允许直接接收任意外部 `project_id` 或 `user_id` 后就按该上下文执行。
- 文档操作统一以 `document_id` 作为外部契约，不再以文件路径作为业务输入。

本阶段明确不做：

- 多租户组织管理
- 租户级管理后台
- 跨租户共享
- 租户之间的数据路由

## 3. 由这些结论推导出的架构约束

以下约束从阶段 1 开始视为强制约束：

### 3.1 前端边界

- React 前端是展示与交互层，不是身份来源。
- 浏览器不能再承担模型密钥或敏感配置透传职责。
- 前端只能在当前用户授权项目范围内切换 active project。
- 后续新增的管理页也必须走同一套服务端权限模型。

### 3.2 API 边界

- 公共 `/api/v1/*` 路由在共享部署场景下默认都需要登录。
- internal 路由必须保持 admin-only，不能退回到“带 header 即内部请求”的模式。
- 权限拒绝必须返回正确 HTTP 状态，并保留审计记录。
- 文档相关接口必须从 API 到服务层都坚持 `document_id` 契约。

### 3.3 存储边界

- 上传文件必须进入受控的持久化文件存储，而不是开发期随手放置的目录。
- 状态数据库必须能跨重启保留会话、消息、审计和任务状态。
- 向量库必须能跨重启保留知识索引结果。
- 隔离区必须与知识库活跃目录物理分离。

### 3.4 配置边界

- 非敏感运行参数可以由 Aurora 持久化。
- 敏感密钥必须由运维注入，不能再由 UI 直接写入 `.env`。
- 任何会暴露到浏览器的配置，都必须默认视为非敏感信息。

## 4. 当前仓库对齐情况

仓库里已经出现了一部分与这套基线对齐的实现：

- `app/auth.py` 已经定义了 `admin / operator / member / viewer` 和 `trusted_header` 模式
- `app/api/dependencies.py` 已经开始集中做认证与权限校验
- `app/services/application_audit_service.py` 已经具备应用级审计落库能力
- `app/api/routes/documents.py` 已经具备上传大小限制、MIME 校验和隔离区处理
- `app/api/routes/knowledge_base.py`、`app/api/routes/logs.py`、`app/api/routes/settings.py` 已经在走路由级权限收口
- 文档接口已经在向 `document_id` 契约迁移

当前仍然阻塞后续阶段推进的主要缺口：

- `frontend/src/api.js` 仍在构造浏览器侧 `X-LLM-API-Key` 一类透传头
- 共享部署推荐值此前没有被明确写进 `.env.example`
- 部分 internal 校验链路仍允许较丰富的上下文覆盖，只能继续保留在 admin-only 范围
- Playwright 还没有形成带 `webServer` 的冷启动验收形态
- 部署产物、备份脚本和恢复脚本尚未补齐

## 5. 分阶段实施路线

### 阶段 1：共享内测门槛

目标：把 Aurora 从“适合单人使用”推进到“适合内网共享内测”。

必须完成：

- 接入登录鉴权，从 SSO/session 中统一拿 `user_id / team / role`
- 删除公共业务接口里客户端自报身份的主路径
- 所有 `internal_*` 路由改成真正的管理员权限
- settings、logs、knowledge-base rebuild/reset、provider dry-run 全部加权限控制
- 浏览器侧移除 `X-LLM-API-Key`、`X-Embedding-API-Key` 等透传模式
- `CORS_ORIGINS` 收紧为内网域名白名单
- 上传接口补齐文件大小上限、MIME 校验和恶意文件隔离策略

出阶段标准：

- 未登录用户和低权限用户无法通过公共路由或 header 技巧触达管理能力

### 阶段 2：运行治理门槛

目标：把 Aurora 从“受保护的应用”推进到“可治理的内部服务”。

必须完成：

- 配置中心拆成可在线修改的非敏感参数和仅运维注入的密钥配置
- UI 不再直接写 `.env`
- 为配置修改、文档删除、知识库重建、权限拒绝补齐应用审计日志
- 文档服务层彻底收敛到 `document_id`，禁止路径级绕行
- 为聊天、上传、重建、日志查询增加限流和并发保护
- 将 `AbuseGuard` 扩展到主链路
- 知识库任务改成可恢复的后台任务模型，保证单实例重启后状态可信

出阶段标准：

- 关键管理动作可追踪、可恢复、可限流

### 阶段 3：试点上线门槛

目标：把 Aurora 从“能内测”推进到“能重复部署并完成试点验收”。

必须完成：

- 增加 API 集成测试，覆盖未登录、越权、跨项目、管理员操作、配置审计、上传限制
- Playwright 使用自带 `webServer` 拉起服务，形成可重复冷启动验收
- 补齐 `Dockerfile` / `docker-compose` 或 `systemd` 部署脚本
- 增加环境模板、备份脚本、恢复脚本
- 增加 health/ready、日志轮转、数据库备份、知识库目录备份
- 拆分 `frontend/src/App.jsx`，避免多角色与管理页继续堆成单文件

出阶段标准：

- 一套命令可以从零拉起系统并跑完 API + E2E 验收

## 6. 试点验收门槛

团队试点上线前，以下条件必须全部满足：

- 未登录用户无法访问任何业务 API
- 普通成员不能改配置、清日志、重建知识库、访问内部接口
- 用户不能伪造别人的 `project / user / session` 上下文来访问数据
- 浏览器里不再出现模型密钥透传
- 上传超限、异常类型、并发重建都有明确拦截
- 所有关键管理动作都有审计记录
- 一套命令能从零拉起并跑完 API + E2E 验收

## 7. 明确不纳入本轮范围的事项

以下内容不在这次路线里，不应被偷偷带入实现假设：

- 跨租户共享
- 租户级品牌化或租户级管理后台
- 公网直连暴露
- 浏览器侧托管 provider 密钥
- UI 直接把密钥写入 `.env`
- 多实例 active-active 调度

## 8. 推荐执行顺序

按照这份基线，最稳妥的推进顺序是：

1. 完成认证收口并移除浏览器密钥透传
2. 收紧管理权限与 internal 路由
3. 补齐审计覆盖和 `document_id` 服务层收口
4. 完成限流、并发保护和重建任务恢复
5. 最后补部署产物、备份恢复和重复验收链路

## 9. 阶段 0 完成定义

阶段 0 完成，不代表所有功能都已实现；它代表后续开发不再以“架构还没定”为由继续分叉。

满足以下条件即可视为阶段 0 完成：

- 关键架构决策已经冻结
- 部署与安全边界已经写清楚
- 后续阶段具备明确进入条件和退出条件
- 未完成项被视为实施任务，而不是开放式架构讨论
