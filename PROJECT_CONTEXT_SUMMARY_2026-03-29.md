## 3. 我对项目的基本了解

### 3.1 项目定位

Aurora 是一个面向软件测试知识场景的本地知识工作台。

它不是单纯的聊天壳子，也不是单纯的文件管理器，而是把下面几类能力串成一个闭环：

- 文档沉淀
- 知识库构建
- RAG 检索问答
- 引用验证
- 知识图谱浏览
- 日志排查
- 模型配置与联调

更具体地说，它适合以下场景：

- 测试团队沉淀内部知识资料
- 用真实资料验证 RAG 检索和回答质量
- 联调模型接口、Embedding、切片参数
- 通过日志和图谱辅助排查知识库构建问题

### 3.2 当前真实技术栈

后端：

- Python 3.11+
- FastAPI
- Uvicorn
- LlamaIndex
- Chroma
- OpenAI / OpenAI Compatible 调用方式

前端：

- React 18
- Vite 7
- 原生 CSS
- 单页控制台实现

### 3.3 当前真实代码结构

后端核心目录：

- `app/server.py`：FastAPI 入口与路由挂载
- `app/api/`：系统、知识库、图谱、日志、设置、对话接口
- `app/services/`：目录扫描、索引构建、RAG、图谱、日志、配置等服务逻辑
- `api_server.py`：本地启动入口

前端核心目录：

- `frontend/src/App.jsx`：当前前端主入口，承担页面编排与主要交互逻辑
- `frontend/src/api.js`：前端请求层
- `frontend/src/lib/document-utils.js`：文档相关工具函数
- `frontend/src/styles.css`：全局样式

### 3.4 一个重要事实

当前真实前端已经是 React 实现，不是旧交接文档里那套 `Vue + 多视图文件` 结构。

因此继续开发时，应以当前仓库实际文件为准，而不是完全依赖旧交接文档。

## 5. 当前未完成的事情

结合 [UNFINISHED_BACKLOG.md](./UNFINISHED_BACKLOG.md)，以及当前代码现状，未完成项主要分为以下几类。

### 5.1 架构层未完成

- `frontend/src/App.jsx` 仍然过大，前端主文件还没有拆分成页面组件、通用组件和 hooks
- 知识库重建任务目前仍偏进程内状态管理，缺少任务持久化
- 文档增量索引能力虽然已有基础，但状态流转与任务编排还可以继续做实

### 5.2 测试层未完成

- 当前以后端 service 测试和前端工具/请求层测试为主
- 缺少 API 集成测试
- 缺少前后端联动的页面级自动化回归

### 5.3 产品层未完成

- 本地演示模式已可用，但产品边界、提示策略、正式模式差异说明还不够清晰
- 多模型接入目前主要围绕 OpenAI / OpenAI Compatible 体系
- 更完整的团队协作能力尚未建设

### 5.4 体验层未完成

- 首页、总览、日志页虽然已经完成一轮压缩，但仍有继续收敛空间
- 一些卡片仍然偏高，信息密度与首屏利用率还可以继续优化
- 个别说明文案仍略多，需继续平衡“可理解”和“不过度解释”

## 7. 我认为项目下一步最值得优化升级的地方

下面这些建议不是安全项，重点是产品、工程、体验三个维度的升级方向。

### 7.1 P1：前端结构拆分

这是当前最值得尽快做的工程项。

原因：

- `frontend/src/App.jsx` 体量已经很大
- 多页面逻辑、状态、弹窗、交互集中在一个文件里，维护成本会越来越高
- 后续再做 UI 精修、页面增强或多人协作时，冲突风险高

建议拆分方向：

- 页面级组件：总览、知识库、对话、图谱、设置、日志
- 通用组件：卡片、标题栏、统计块、弹窗、空状态、工具按钮
- hooks：数据加载、日志筛选、知识库任务轮询、运行时配置

### 7.2 P1：API 集成测试补齐

当前功能可用，但还缺少“从 HTTP 接口层出发”的真实回归。

建议优先覆盖：

- 文档上传
- 文档元数据修改
- 重建知识库
- 获取图谱
- 设置保存与测试
- 对话问答
- 日志筛选

### 7.3 P1：知识库任务体系做实

目前 Aurora 已经有“可重建”的能力，但未来真正稳定迭代时，任务体系需要更强。

建议补强：

- 重建任务持久化
- 任务恢复策略
- 重复重建保护
- 失败状态标记与重试入口
- 更明确的任务阶段划分

### 7.4 P1：继续做信息密度优化

当前 UI 已经从“重说明”往“重状态”转，但还没有完全到位。

建议后续继续优化：

- 首页 hero 再精简
- 总览页卡片再压扁
- 日志页进一步减少表单高度
- 标题、副标题、说明文字统一分级
- 把真正高频动作更靠近首屏

### 7.5 P2：本地演示模式产品化

这是 Aurora 很有价值的一点，但目前更像工程降级方案，不够像成熟产品能力。

建议继续完善：

- 清晰定义“演示模式”和“正式模式”的功能差异
- 给出更明确的 UI 状态标记
- 增加更有说服力的演示问答策略
- 在 README 和页面内给出一致说明

### 7.6 P2：多模型能力继续扩展

当前已内置多厂商预设，但后端主路径仍以 OpenAI Compatible 为主。

建议后续扩展：

- Ollama
- Anthropic
- Gemini
- 更明确地区分聊天模型与 Embedding 模型能力

### 7.7 P2：图谱与日志页做更深的产品化

图谱页和日志页已经可用，但仍偏“工程工具感”。

建议方向：

- 图谱增加筛选记忆、节点聚焦、引用跳转
- 日志页增加常用筛选预设
- 日志页增加错误聚合与摘要
- 图谱与对话、图谱与文档之间做更多双向联动

## 9. 继续接手时的建议顺序

建议下一个开发阶段按下面顺序推进：

1. 先做 `App.jsx` 拆分
2. 再补 API 集成测试
3. 再做知识库任务持久化与增量任务编排
4. 然后继续做前端信息密度与交互精修
5. 最后再扩展更多模型与演示模式产品化

## 10. 一句话总结

Aurora 当前已经不是“缺主流程的原型”，而是一个主链路完整、可演示、可继续产品化和工程化升级的软件测试知识工作台；下一阶段最重要的不是再堆功能，而是把结构、测试、任务体系和信息密度继续做扎实。
