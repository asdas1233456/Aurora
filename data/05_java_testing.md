# Java 测试经验与实践

本文整理 Java 项目中常见的测试方法、框架使用经验和注意事项。

## 1. Java 测试分层思路

### 单元测试

常用框架：

- JUnit 5
- Mockito
- AssertJ

目标：

- 验证方法级逻辑
- 验证业务规则
- 验证参数与异常

### 接口测试

常见工具：

- RestAssured
- Postman
- JMeter

目标：

- 验证服务接口行为
- 验证协议、状态码、数据结构、鉴权逻辑

### 集成测试

目标：

- 验证 Spring Boot、数据库、缓存、中间件协作

## 2. 常用框架经验

### JUnit 5

优点：

- 注解清晰
- 生命周期更清楚
- 与现代 Java 生态适配更好

常见注解：

- `@Test`
- `@BeforeEach`
- `@AfterEach`
- `@ParameterizedTest`

### Mockito

适合场景：

- 替代外部依赖
- 构造异常路径
- 验证方法调用次数和参数

经验：

- Mock Repository、外部 API 客户端比较常见
- 不建议把业务逻辑本身 Mock 掉

## 3. Spring Boot 测试经验

### 分层测试

- Controller 层：重点测参数校验、返回码、返回结构
- Service 层：重点测业务逻辑
- Repository 层：重点测数据库交互

### 常见注解

- `@SpringBootTest`
- `@WebMvcTest`
- `@DataJpaTest`
- `@MockBean`

经验：

- `@SpringBootTest` 功能强，但启动慢
- 能局部测试就尽量不要每次都全量启动 Spring 容器

## 4. 重点测试点

### 参数校验

需要覆盖：

- 必填字段缺失
- 字段长度超限
- 枚举值非法
- 数字边界
- 时间格式异常

### 异常处理

需要覆盖：

- 业务异常
- 系统异常
- 数据不存在
- 重复数据
- 并发更新冲突

### 数据一致性

需要覆盖：

- 提交事务是否成功
- 回滚是否生效
- 多表更新是否一致

## 5. Java 接口测试经验

### 断言不要只校验 success 字段

还应关注：

- HTTP 状态码
- code
- message
- data 结构
- 数据内容是否符合业务规则

### 复杂接口要覆盖鉴权场景

例如：

- 未登录访问
- token 失效
- 无权限角色访问
- 越权访问

## 6. 常见问题

### 测试环境依赖太重

问题：

- 启动慢
- 用例执行慢
- 环境不稳定

建议：

- 单元测试尽量轻量化
- 集成测试只覆盖关键链路
- 不要所有测试都依赖完整环境

### 断言写得太弱

错误示例：

- 只断言返回不为空
- 只断言 `200`

更好的做法：

- 断言具体业务字段
- 断言数据库结果
- 断言副作用是否发生

## 7. Java 测试建议

- 单元测试优先验证业务规则和边界条件
- 接口测试优先覆盖权限、异常和数据一致性
- 回归测试要沉淀为可复用的数据驱动用例
- 测试代码也要重构，不要只重构业务代码
