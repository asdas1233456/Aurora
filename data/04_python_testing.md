# Python 测试经验与实践

本文整理 Python 项目中常见的测试思路、分层方法和实战经验。

## 1. Python 测试分层

### 单元测试

目标：

- 验证函数、类、模块是否按预期工作
- 尽量隔离外部依赖

适合测试：

- 工具函数
- 数据处理逻辑
- 参数校验
- 业务规则

### 接口测试

目标：

- 验证 API 输入输出
- 验证状态码、响应结构、异常处理

### 集成测试

目标：

- 验证多个模块之间协作是否正确
- 验证数据库、缓存、消息队列等依赖

## 2. 常见测试框架

### pytest

优点：

- 语法简洁
- fixture 机制强大
- 插件生态丰富
- 适合单元测试、接口测试、UI 自动化测试

常见用法：

- `assert`
- `fixture`
- `parametrize`
- `mark`

经验：

- 新项目优先推荐 pytest
- 测试数据准备和资源回收尽量放到 fixture 中

### unittest

优点：

- Python 标准库自带
- 和传统 xUnit 风格一致

适合场景：

- 老项目维护
- 对标准库依赖要求高的场景

## 3. 重点测试点

### 参数边界

需要覆盖：

- 空值
- None
- 超长字符串
- 特殊字符
- 非法类型
- 极值

### 异常分支

需要覆盖：

- 输入不合法
- 外部服务异常
- 数据库操作失败
- 文件不存在
- 权限不足

### 状态变化

需要覆盖：

- 创建成功后状态是否正确
- 更新后是否持久化
- 删除后是否不可见

## 4. Mock 的使用经验

### 适合 Mock 的对象

- 第三方接口
- 数据库访问
- 文件系统
- 时间函数
- 随机数

### 不要过度 Mock

问题：

- 用例能过，但真实场景不一定能过
- 测试变成“验证 Mock 自己”

建议：

- 单元测试可以多用 Mock
- 集成测试要减少 Mock，尽量接近真实环境

## 5. 接口测试经验

### 核心断言不要只看状态码

应该同时校验：

- 状态码
- 响应字段
- 字段类型
- 关键业务值
- 错误码
- 错误信息

### 负向用例要覆盖

例如：

- 缺少参数
- 非法参数
- 重复提交
- 无权限访问
- 资源不存在

## 6. 目录结构建议

```text
project/
├─ app/
├─ tests/
│  ├─ unit/
│  ├─ integration/
│  ├─ api/
│  └─ conftest.py
```

经验：

- 测试目录按类型分层
- 公共 fixture 放在 `conftest.py`
- 测试数据和测试代码分开管理

## 7. 常见问题

### 用例不稳定

原因可能是：

- 依赖真实时间
- 依赖执行顺序
- 依赖共享数据
- 并发冲突

解决思路：

- 固定测试数据
- 每个用例独立准备环境
- 避免依赖上一个用例的结果

### 用例很多但质量不高

表现：

- 覆盖率看起来不错
- 实际线上问题依旧频繁

原因：

- 只测主流程
- 没测边界
- 没测异常
- 断言过弱

## 8. Python 测试建议

- 优先写高价值用例，而不是堆数量
- 对核心逻辑，单元测试要覆盖到异常分支
- 对接口测试，要重点关注错误处理和数据一致性
- 对异步任务、定时任务、重试逻辑要单独设计测试
