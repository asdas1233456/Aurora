# Python 测试经验与实践

本文整理 Python 项目里常见的测试分层、工具选择、断言策略和稳定性经验，适合接口服务、脚本工具和自动化工程场景。

## 1. Python 测试分层

### 单元测试

常用工具：
- pytest
- unittest
- pytest-mock
- freezegun

重点目标：
- 验证函数级逻辑
- 验证边界条件
- 验证异常路径
- 验证输入输出转换

### 集成测试

适用场景：
- FastAPI / Flask 接口测试
- 数据库读写
- Redis、消息队列、对象存储联动
- 外部服务适配层验证

重点目标：
- 验证模块之间协作
- 验证配置和依赖注入是否正确
- 验证事务与回滚逻辑

### 端到端测试

适用场景：
- 用户从页面或接口发起完整业务流程
- 多服务链路联调
- 发布前回归

重点目标：
- 验证核心业务闭环
- 验证真实环境配置
- 验证关键依赖稳定性

## 2. pytest 常用实践

### fixture 设计

建议：
- fixture 只负责准备数据和依赖
- 避免一个 fixture 做太多事
- 对昂贵资源设置合理 scope
- 用工厂 fixture 减少重复数据构造

常见 scope：
- function：最安全，隔离性最好
- module：适合同模块共享资源
- session：适合测试数据库、临时目录、应用实例

### 参数化测试

适用场景：
- 校验不同输入组合
- 校验边界值
- 校验格式兼容性

建议：
- 用例名要可读
- 正常值和异常值分开组织
- 不要把几十个场景硬塞进一个参数化函数里

## 3. FastAPI / Flask 接口测试经验

重点关注：
- 状态码是否正确
- 返回结构是否稳定
- 鉴权是否生效
- 错误信息是否可读
- 非法参数是否被拦截

建议断言：
- 不只断言 200
- 断言业务字段和值
- 断言数据库副作用
- 断言日志或事件是否产生

## 4. Mock 使用原则

适合 Mock 的对象：
- 第三方 HTTP 客户端
- 消息发送器
- 邮件、短信、推送服务
- 时钟、UUID、随机数

不建议过度 Mock 的对象：
- 核心业务逻辑本身
- 领域规则判断
- 本来就很轻的内部函数

经验：
- Mock 太多会让测试通过但业务还是坏的
- 真正要稳定的是依赖边界，不是把整个系统都替身化

## 5. 数据库测试经验

建议覆盖：
- 新增成功
- 更新成功
- 唯一约束冲突
- 事务回滚
- 并发更新冲突
- 软删除与查询过滤

建议：
- 用独立测试库或临时数据库
- 每条用例尽量自清理
- 对查询结果不要只看数量，也要看具体字段

## 6. 常见问题

### 测试慢

原因常见于：
- 启动整个应用太频繁
- 访问真实网络
- 数据准备太重
- 重复初始化数据库

优化建议：
- 单元测试优先轻量化
- 集成测试只覆盖关键链路
- 用 fixture 复用应用实例

### 测试不稳定

原因常见于：
- 依赖真实时间
- 依赖随机数
- 依赖执行顺序
- 共用脏数据

优化建议：
- 固定时间和随机值
- 每条用例独立准备数据
- 避免跨用例共享可变状态

## 7. Python 测试建议

- 优先用 pytest 统一测试风格
- 单元测试强调边界和异常，不只测 happy path
- 接口测试同时验证响应和副作用
- 对关键 bug 补回归用例，避免只修不测
- 测试代码也要重构，保持可读性和低重复
