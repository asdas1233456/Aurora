# 缺陷生命周期与 Bug 报告规范

本文整理缺陷从发现到关闭的常见流程，以及一份高质量 Bug 报告应该具备的内容。

## 1. 缺陷生命周期

常见状态：
- New
- Assigned
- Open / Processing
- Fixed
- Ready for Test
- Reopened
- Closed
- Rejected

不同团队命名可能不同，但本质流程相近。

## 2. 什么样的 Bug 报告才算合格

至少应包含：
- 标题
- 环境信息
- 前置条件
- 复现步骤
- 实际结果
- 预期结果
- 影响范围
- 附件证据

## 3. 标题怎么写

好的标题特点：
- 具体
- 可检索
- 能体现模块和现象

示例：
- 用户管理页删除员工后列表未刷新
- 登录接口 token 过期时返回 500 而不是 401

## 4. 复现步骤怎么写

建议：
- 一步一行
- 不省略关键前置条件
- 能让开发独立复现

不要写成：
- “正常操作后报错”

应该写成：
1. 使用普通角色账号登录后台
2. 进入用户管理页
3. 搜索员工 A
4. 点击删除并确认
5. 观察列表区域

## 5. 结果描述怎么写

实际结果：
- 点击删除后弹窗关闭，但列表仍展示已删除员工

预期结果：
- 删除成功后列表自动刷新，不应再展示该员工

## 6. 严重程度与优先级

严重程度通常描述影响：
- 致命
- 严重
- 一般
- 轻微

优先级通常描述修复紧急度：
- P0
- P1
- P2
- P3

注意：
- 严重程度不一定等于优先级
- 视觉问题在大促首页也可能是高优先级

## 7. 常见低质量 Bug 报告问题

- 没有复现步骤
- 没有环境信息
- 只有截图没有现象描述
- 标题太泛
- 预期结果不明确

## 8. 测试与开发协作建议

- 提交 Bug 前先做最基本复查
- 能定位到接口、日志、请求体时尽量补上
- 对偶现问题保留时间、账号、设备、请求 ID
- 修复后优先回归主流程和相邻场景

## 9. Bug 管理建议

- 每个高优先级缺陷都应有明确 owner
- 对反复出现的问题要追溯根因
- 发布前关注 reopened 和未验证缺陷
- 缺陷库本身也是知识库，应可复用、可检索
