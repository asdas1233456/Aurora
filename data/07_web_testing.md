# Web 测试经验

本文整理 Web 项目测试中常见的功能、兼容性、可用性与自动化经验。

## 1. Web 功能测试重点

### 页面级验证

重点关注：

- 页面是否能正常打开
- 资源是否完整加载
- 文案是否正确
- 图片、按钮、链接是否可用

### 表单类验证

重点关注：

- 必填校验
- 格式校验
- 长度校验
- 特殊字符
- 重复提交
- 错误提示是否清晰

### 列表与搜索验证

重点关注：

- 分页
- 排序
- 筛选
- 搜索结果正确性
- 空数据展示

## 2. 浏览器兼容性测试

常见关注点：

- Chrome
- Edge
- Firefox
- Safari

重点问题：

- CSS 样式差异
- 字体与布局偏差
- JS 兼容性问题
- 文件上传下载兼容问题

经验：

- 不同浏览器的测试重点不同
- 管理后台常以 Chrome 为主，面向公众的网站要更重视兼容性

## 3. 前端异常测试

需要关注：

- 接口超时
- 接口返回空值
- 接口返回错误码
- JS 报错
- 静态资源加载失败

经验：

- 很多页面不是“功能错了”，而是“异常情况下无反馈”
- Web 测试应配合浏览器 F12 的 Network 和 Console 一起看

## 4. 页面自动化测试经验

### 适合自动化的场景

- 登录
- 核心下单流程
- 关键配置流程
- 稳定的 CRUD 场景

### 不适合优先自动化的场景

- 页面频繁改版
- 交互非常不稳定
- 一次性活动页

### 编写自动化时的建议

- 优先使用稳定定位方式
- 不要依赖脆弱的绝对 XPath
- 等待条件要明确
- 断言要和业务结果绑定

## 5. Web 测试常见问题

### 元素定位失败

原因可能是：

- 页面未加载完成
- iframe 未切换
- 元素在 Shadow DOM 中
- 动态渲染

### 自动化脚本不稳定

原因可能是：

- 等待不足
- 网络抖动
- 测试数据污染
- 环境依赖不稳定

## 6. Web 测试建议

- 手工测试先把业务流和异常流跑清楚，再做自动化
- 接口测试和前端测试要联动，不要只看页面表现
- Bug 提交时最好附带：
  - 页面地址
  - 浏览器版本
  - 操作步骤
  - 控制台错误
  - 网络请求信息
