# 性能测试经验

本文整理性能测试中常见的指标、方法、压测思路和问题定位经验。

## 1. 性能测试的常见目标

### 响应时间

关注：

- 平均响应时间
- P90 / P95 / P99
- 最大响应时间

### 吞吐量

关注：

- TPS
- QPS
- 并发用户数

### 资源利用率

关注：

- CPU
- 内存
- 磁盘 IO
- 网络 IO

## 2. 常见性能测试类型

### 基准测试

目的：

- 测量系统在正常负载下的表现

### 负载测试

目的：

- 验证系统在预期并发下是否满足 SLA

### 压力测试

目的：

- 找出系统瓶颈点和崩溃点

### 稳定性测试

目的：

- 验证系统长时间运行是否稳定

## 3. 性能测试设计思路

### 明确业务场景

不要只压一个接口，要先搞清楚：

- 核心交易链路是什么
- 用户最常用的操作是什么
- 哪些接口是高频接口
- 哪些接口最消耗资源

### 设计真实比例

例如：

- 登录 10%
- 查询列表 50%
- 查询详情 25%
- 提交操作 15%

经验：

- 真实业务比例比“单接口打爆”更有价值

## 4. 常见工具

### JMeter

适合：

- HTTP 接口压测
- 参数化
- 断言
- 简单分布式压测

### Locust

适合：

- 使用 Python 编写压测逻辑
- 构造复杂用户行为

### k6

适合：

- 脚本化性能测试
- CI 中集成

## 5. 常见瓶颈点

### 应用层

- 线程池不足
- 锁竞争
- 代码效率低
- 同步阻塞严重

### 数据库层

- 慢 SQL
- 索引缺失
- 连接池不足
- 大事务

### 中间件层

- 缓存击穿
- MQ 堆积
- 网关限流

### 基础设施层

- CPU 打满
- 内存泄漏
- 磁盘 IO 高
- 网络拥塞

## 6. 性能测试中的观察点

不仅要看压测工具结果，还要同时看：

- 服务端日志
- 数据库慢日志
- JVM 指标
- 容器资源
- 系统负载
- 错误率

## 7. 常见误区

### 误区 1：只看平均响应时间

问题：

- 平均值容易掩盖长尾问题

正确做法：

- 一定要看 P95、P99

### 误区 2：只压接口，不看业务

问题：

- 压测结果和真实线上差距大

### 误区 3：压出问题后只怪开发

正确思路：

- 性能问题通常是系统问题，不是某一个人的问题
- 要从架构、数据库、代码、配置、资源一起看

## 8. 性能测试建议

- 压测前先确定目标值和验收标准
- 压测中要同步监控服务端指标
- 压测后要有完整结论：
  - 达没达到目标
  - 瓶颈在哪里
  - 风险是什么
  - 优化建议是什么

## 9. Bug 与报告建议

性能问题报告中建议包含：

- 测试环境说明
- 压测场景说明
- 并发模型
- 压测时长
- 指标结果
- 错误率
- 服务端资源曲线
- 结论与建议
