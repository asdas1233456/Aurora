# 性能测试方法与实战要点

本文整理性能测试中常见的指标、场景设计、压测执行方式和问题分析思路，适合接口服务、Web 系统和业务后台。

## 1. 性能测试关注什么

常见目标：
- 响应时间是否达标
- 并发量是否可承受
- 吞吐量是否稳定
- 系统是否存在资源瓶颈
- 长时间运行是否出现性能衰减

## 2. 常见性能指标

重点指标：
- 平均响应时间
- P90 / P95 / P99 响应时间
- TPS / QPS
- 并发用户数
- CPU 使用率
- 内存使用率
- 磁盘 IO
- 网络带宽
- 错误率

经验：
- 不要只看平均响应时间
- 高分位延迟往往更能说明真实体验问题
- 错误率和超时率必须结合一起看

## 3. 性能测试场景设计

### 基准测试

目标：
- 获取系统在低压下的稳定基线
- 观察单请求响应时间

### 负载测试

目标：
- 验证目标并发和业务量下是否稳定
- 判断系统能否达到预期容量

### 压力测试

目标：
- 找到系统拐点
- 观察资源耗尽时的表现

### 稳定性测试

目标：
- 验证长时间运行后的资源泄漏
- 观察线程、连接、缓存是否异常增长

### 峰值测试

目标：
- 模拟活动流量、任务高峰、批量提交
- 验证突刺流量是否会打垮系统

## 4. 压测前准备

建议先确认：
- 测试环境是否接近生产
- 测试数据是否有代表性
- 日志级别是否合理
- 监控是否齐全
- 限流、熔断、缓存策略是否已知

如果这些前提不清楚，压测结果通常没有解释力。

## 5. 常见瓶颈定位思路

### 接口慢

优先排查：
- SQL 是否慢
- 索引是否缺失
- 外部接口是否阻塞
- 应用线程池是否耗尽
- 缓存是否失效

### CPU 高

优先排查：
- JSON 序列化过重
- 大量循环或正则
- 频繁对象创建
- 加解密与压缩计算

### 内存高

优先排查：
- 缓存未淘汰
- 大对象堆积
- 列表一次性加载过多
- 连接或会话未释放

## 6. Web 性能测试关注点

前端侧重点：
- 首屏时间
- 静态资源大小
- 接口 waterfall
- 页面交互卡顿
- 大表格和长列表渲染

后端侧重点：
- 接口响应时间
- 慢查询
- 网关与鉴权链路
- 文件上传下载

## 7. 压测结果怎么写

建议至少包含：
- 测试目标
- 环境说明
- 数据准备方式
- 场景说明
- 关键指标结果
- 瓶颈分析
- 风险结论
- 优化建议

不要只给一张图或一句“通过/不通过”。

## 8. 常见误区

- 只压接口，不看数据库和系统资源
- 压测环境和生产差异过大
- 不区分预热和正式数据
- 只看峰值，不看长稳
- 没有错误日志和监控还做结论

## 9. 性能测试建议

- 先做基准，再做负载，再做压力
- 指标要和业务 SLA 对齐
- 压测一定结合监控、日志、慢 SQL 一起看
- 压测结论要能指导优化，不只是做结果展示
- 每次优化后要回归，确认是否真的提升
