# 需求评审与测试分析

本文整理测试在需求阶段应该重点关注的问题，帮助团队把问题前移，而不是等开发完成后再集中爆炸。

## 1. 测试为什么要参加需求评审

测试的价值不只是执行用例，还包括：
- 发现需求歧义
- 发现遗漏场景
- 提前识别风险
- 推动可测性设计

## 2. 评审时重点看什么

重点关注：
- 业务规则是否完整
- 状态流转是否清楚
- 边界条件是否明确
- 异常处理是否定义
- 权限规则是否明确
- 兼容性要求是否明确
- 埋点、日志、监控是否考虑

## 3. 常见需求歧义

典型问题：
- “支持批量操作”但没写上限
- “失败时提示错误”但没写提示内容
- “管理员可见”但没写具体角色
- “导出成功”但没写导出字段和格式

## 4. 测试分析输出建议

建议沉淀：
- 关键场景清单
- 异常场景清单
- 风险点
- 待确认问题
- 建议新增的技术可观测性

## 5. 可测性建议

可推动开发补充：
- 稳定的错误码
- 操作日志
- 请求 ID
- 状态字段
- 可查询后台数据

## 6. 实战建议

- 需求评审时优先问清规则，不急着写用例
- 对不明确的问题留痕，不要靠口头记忆
- 复杂需求建议同步产出流程图或判定表
